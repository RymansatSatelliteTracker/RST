# RST アーキテクチャ

## システム概要

RST (Rymansat Satellite Tracker) は、衛星追跡・アンテナ制御・無線機制御を統合したElectronベースのデスクトップアプリケーションです。

- **フロントエンド**: Vue 3 + TypeScript + Vuetify
- **バックエンド**: Node.js + TypeScript (Electronメインプロセス)
- **ターゲットOS**: Windows / macOS / Linux

---

## プロセス構造

Electronの2プロセス構成（メインプロセス + レンダラープロセス）に、IPC通信ブリッジを介して連携します。

```
┌─────────────────────────────────────────────────────────┐
│                   Electron App                          │
│                                                         │
│  ┌──────────────────┐    IPC     ┌──────────────────┐  │
│  │   Main Process   │◄──────────►│Renderer Process  │  │
│  │  (Node.js)       │  preload   │  (Vue 3 SPA)     │  │
│  │                  │            │                  │  │
│  │  サービス群      │            │  コンポーネント群│  │
│  │  ハードウェア制御│            │  衛星計算        │  │
│  │  ファイルI/O     │            │  地図表示        │  │
│  └──────────────────┘            └──────────────────┘  │
│           │                                             │
│  ┌────────┴────────┐                                    │
│  │  外部デバイス    │                                    │
│  │  ・ローテーター  │                                    │
│  │  ・無線機        │                                    │
│  └─────────────────┘                                    │
└─────────────────────────────────────────────────────────┘
```

## レイヤ構成

### 呼び出し階層の基本パターン

```
[ Renderer Process ]

  Vue Component (organisms / molecules)
         |  uses
         v
  Composable  (useXxx)
         |  calls
         v
  Renderer Service
         |  window.rstApi.xxx()
         v
  IPC Bridge  (preload / contextBridge)
         |  ipcRenderer.invoke / ipcRenderer.on
         v
[ Main Process ]

  IPC Handler  (ipcMain.handle)
         |  calls
         v
  Main Service
         |
         +---> Hardware Controller  (Serial)
         +---> File I/O  (electron-store / JSON)
         \---> External API  (HTTP)
```

### フロントエンド（レンダラープロセス）

コンポーネントはAtomic Designの考え方で3層に分類します。

```
atoms/      ... 単独で意味を持つ最小UI部品（ボタン、テキストフィールドなど）
molecules/  ... atoms を組み合わせた複合UI部品（選択コンポーネントなど）
organisms/  ... 独立した機能を持つUIブロック（地図、制御パネルなど）
```

UIロジックはComposable（`useXxx`）に切り出し、コンポーネントとサービス層の橋渡しをします。

### 無線機制御Composableの責務分割

`TransceiverCtrl` 配下では、`useTransceiverCtrl` のステップ増大を抑えるために以下の責務分割を行っている。

| クラス/Composable                   | 主責務                                                                                                                    |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `useTransceiverCtrl`                | UI状態の公開、watch登録、各クラスのオーケストレーション（watch本体は薄く保ち、詳細処理はヘルパー関数や責務クラスへ委譲）  |
| `TransceiverBaseFreqMgr`            | Tx/Rx基準周波数（補正値なし）の保持、補正値反映後の基準周波数算出                                                         |
| `TransceiverFreqCoordinator`        | 周波数初期化、無線機への周波数送信、（内部保持する `TransceiverDopplerCalc` による）ドップラー補正更新                    |
| `TransceiverModeCoordinator`        | Auto/Beacon/Satellite のモード遷移シーケンス管理、状態退避・復元、更新タイマー制御、NoradId 同期・変更判定                |
| `TransceiverModeStateResolver`      | サテライトモード切替時の状態保存/復元、`isSatelliteMode` とトラッキングモードの反映                                       |
| `TransceiverDopplerWaitCoordinator` | ドップラーシフト待機状態と待機タイマの管理（待機開始/延長/解除）                                                          |
| `TransceiverModeSettingResolver`    | Auto/Beaconの状態に応じた周波数・運用モードの解決                                                                         |
| `TransceiverOpeModeResolver`        | 無線機から受信した運用モードのバリデーション、Tx/Rx運用モード状態への反映                                                 |
| `TransceiverRecvFreqResolver`       | 無線機から受信した周波数の反映、（内部保持する `TransceiverDopplerCalc` による）AutoOn時の基準周波数再算出、通知/ログ出力 |
| `TransceiverSyncCoordinator`        | サテライトモードOFF時のTx/Rx周波数・運用モード同期と無限更新防止                                                          |
| `TransceiverDopplerCalc`            | ドップラーシフト有効範囲の判定、ドップラー補正後周波数の算出、無線機周波数変更時の逆ヘテロダイン再算出                    |

`TransceiverModeCoordinator` では、ActiveSatServiceHub からの無線設定取得を private ヘルパーへ集約し、同一呼び出しの重複を避ける。
また、事前条件チェック時の通知付き失敗は private ヘルパーへ集約し、分岐の見通しを維持する。

### バックエンド（メインプロセス）

ハードウェアデバイスはシングルトンサービスが管理し、コントローラーを差し替え可能な構造（Strategyパターン）です。

```
HardwareService  (singleton)
    └── controller/
            ├── 実機コントローラー  (機種ごとに実装)
            └── シミュレーターコントローラー
```

## IPC通信

メインプロセスとレンダラープロセス間はElectron IPCで通信します。`preload.ts` が `contextBridge` 経由で `window.rstApi` としてレンダラーに公開します。

```
Renderer (Vue)
    |
    |  window.rstApi.xxx()
    v
preload.ts  (contextBridge)
    |
    |  ipcRenderer.invoke() / ipcRenderer.on()
    v
Main Process  (ipcMain.handle)
    |
    v
Main Service
```

通信には2種類のパターンがあります。

| パターン        | 方向                                | 用途                       |
| --------------- | ----------------------------------- | -------------------------- |
| invoke / handle | Renderer → Main（同期的）           | データ取得・操作の要求     |
| send / on       | Main → Renderer（非同期・イベント） | ハードウェア状態変化の通知 |

## データフロー

### 衛星追跡フロー

```
外部軌道要素ソース (URL: TLE/3LE/2LE/XML/KVN/JSON/CSV)
    |  HTTP取得
    v
OmmUtil.detectFormat / parseToOmmItems (Main)
    |  形式自動判別 -> OmmItem[]
    v
OMM管理サービス (Main)
    |  omm.json 保存 / IPC (OmmUtil.ommItemToTleStrings で TLE文字列化)
    v
衛星計算サービス (Renderer)
    |  位置・仰角・方位角を計算
    v
アクティブ衛星ハブ (Renderer)
    |
    +---> 地図 / レーダー表示
    +---> ドップラー周波数計算
    \---> アンテナ自動追跡
```

### アンテナ制御フロー

```
アンテナ自動追跡サービス (Renderer)
    |  IPC: 目標方位角・仰角を送信
    v
ローテーターサービス (Main)
    |  シリアル通信
    v
ローテーター本体
    |  現在位置フィードバック
    v
IPC イベント通知
    |
    v
アンテナ制御パネル (Renderer)
```

### 無線機制御フロー

```
ドップラー周波数計算サービス (Renderer)
    |  IPC: 補正周波数を送信
    v
無線機サービス (Main)
    |  CI-V通信 (シリアル)
    v
無線機本体
    |  現在周波数・モードフィードバック
    v
IPC イベント通知
    |
    v
無線機制御パネル (Renderer)
```

## 設定・データ永続化

アプリケーションの設定とデータはユーザーディレクトリに保存されます（Windows: `%APPDATA%/rst/`）。

| データ       | 形式                  | 備考                               |
| ------------ | --------------------- | ---------------------------------- |
| アプリ設定   | JSON (electron-store) | ローテーター・無線機・表示設定など |
| 軌道要素データ(OMM) | JSON (`omm.json`)    | 起動時に外部URLから取得し形式自動判別(TLE/3LE/2LE/XML/KVN/JSON/CSV)した結果を保存。旧`tle.json`は初回起動時に移行 |
| 周波数データ | JSON                  | 衛星ごとの送受信周波数             |
| ログ         | テキスト              | `logs/rst.log`                     |

## 開発環境設定（Copilot）

ワークスペース設定 `.vscode/settings.json` の `chat.tools.terminal.autoApprove` で、GitHub Copilot のターミナル実行時に以下を自動承認する。

- `npx prettier`
- `npx prettier *`
- 正規表現 `^npx prettier(\b.*)?$`（オプション付き呼び出しを含む）

この設定により、`npx prettier` 実行時の承認ダイアログを省略できる。
